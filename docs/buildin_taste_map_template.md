## 1. Problem / Opportunity

### 1.1 Проблема

- Ручные пользовательские теги дают слабую персонализацию.
- Пользователь редко поддерживает теги в актуальном состоянии.
- Рекомендации слабо адаптируются к реальному поведению.

### 1.2 Возможность

- Собрать стартовый вкус-профиль на регистрации.
- Динамически дополнять профиль из отзывов и действий.
- Показать прозрачные гипотезы во вкладке профиля и дать контроль пользователю.

---

## 2. Product Goal

### 2.1 Цель

Сделать персонализацию рекомендаций через профиль вкуса пользователя, который строится из onboarding + фактического поведения.

### 2.2 North Star impact

- Рост `Card -> Route` conversion.
- Рост повторных визитов D7/D30.
- Рост полезности рекомендаций (CTR в персональные блоки).

## 3. Scope

### 3.1 P0 (MVP)

- Онбординг "Карта вкусов" при регистрации (6-10 вопросов).
- Автогенерация `user_taste_tags` на основе ответов.
- Фоновый пересчет вкуса из отзывов (ежедневно/по событию).
- Экран в профиле `Карта вкусов`.
- Блок "Наши предположения" с действиями `Оставить` / `Скрыть`.
- Рекомендации с учетом `user_taste_tags`.

### 3.2 P1

- Кнопка "Пройти заново" (новая версия опроса, без потери истории).
- Explainability: "Почему рекомендуем эту кофейню".
- Версионирование формулы inference (`taste_inference_v1`).

### 3.3 Out of scope (v1)

- Полная ML-модель и real-time ranking.
- Социальное сравнение вкусов между пользователями.
- Push-кампании на основе taste-сигналов.

---

## 4. User Flow

1. Регистрация.
2. Онбординг "Карта вкусов" (выбор предпочтений).
3. Создание базового профиля вкуса.
4. Пользователь оставляет отзывы.
5. Фоновый inference добавляет/обновляет вкус-теги и гипотезы.
6. В профиле пользователь открывает "Карту вкусов".
7. Видит базовые предпочтения + предположения и управляет ими.
8. Discovery показывает персонализированную выдачу.

---

## 5. UX Requirements

- Вкусовая карта должна заполняться < 60 секунд.
- У каждого предположения нужен источник: `опрос`, `отзывы`, `смешанный`.
- "Не нравится X" показывать только при достаточной уверенности.
- Любая гипотеза должна удаляться пользователем.
- Повторный onboarding должен быть безопасным: новая версия, без разрушения предыдущих данных.

## 6. Domain Model (high-level)

| Сущность | Назначение |
|---|---|
| `taste_taxonomy` | словарь вкусовых атрибутов |
| `taste_onboarding_sessions` | ответы пользователя по версиям онбординга |
| `user_taste_profile` | агрегированный профиль пользователя |
| `user_taste_tags` | текущие активные теги со score/confidence |
| `taste_hypotheses` | авто-предположения и их статус |
| `taste_inference_runs` | лог пересчетов для отладки |

---

## 7. Business Rules (v1)

1. Профиль строится из двух источников: onboarding и поведение.
2. Каждый тег хранит `score` и `confidence`.
3. Авто-добавление тега возможно только при `confidence >= threshold`.
4. Авто-негативные гипотезы включаются только при достаточном количестве сигналов.
5. Удаленная пользователем гипотеза не возвращается N дней (cooldown).
6. Ручные действия пользователя имеют повышенный вес в пересчете.

| `taste_profile_recomputed` | завершен inference | `user_id`, `run_id`, `changed_tags_count` |

## 9. Rollout Plan

| Этап | Что включаем | Guardrail |
|---|---|---|
| Stage 0 | internal/staff only | нет деградации по `card->route` |
| Stage 1 | 10% новых пользователей | error rate < 1%, crash free stable |
| Stage 2 | 50% новых + 10% существующих | retention не падает |
| Stage 3 | 100% | целевые метрики подтверждены |

---

## 10. Risks & Mitigations

| Риск | Что делаем |
|---|---|
| Ошибочные гипотезы раздражают | показываем confidence + быстрый dismiss |
| Переусложненный onboarding | 6-10 вопросов, прогресс-бар |
| Накрутка через отзывы | учитываем trusted signals и anti-fraud |
| Низкая объяснимость | блок "Почему мы так думаем" |
