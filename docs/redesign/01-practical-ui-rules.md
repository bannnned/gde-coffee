# Этап 0: UX/UI Правила (Practical UI + Shadcn Direction)

## Назначение документа
Единый, практический и проверяемый свод правил для редизайна.  
Логика выбора решений: `польза и ясность > визуальный шум > вкусовщина`.

## Базовый манифест
1. Дизайн — это система решений, а не набор красивых приемов.
2. Каждый визуальный элемент обязан объяснять структуру, смысл или действие.
3. Чем меньше лишнего, тем быстрее пользователь понимает интерфейс.
4. Консистентность важнее креативных исключений.
5. Доступность не отдельная задача, а стандарт по умолчанию.

## 1. Иерархия и фокус
1. На каждом экране есть один главный action.
2. Визуальный приоритет: `главное -> вторичное -> вспомогательное`.
3. В пределах первого экрана пользователь понимает:
   - где он,
   - что здесь главное,
   - что будет после нажатия.
4. Не конкурируем акцентами: если все “важно”, значит ничего не важно.

## 2. Less is more (упрощение)
1. Удаляем элементы, которые:
   - не помогают выполнить задачу,
   - не уточняют контекст,
   - не влияют на принятие решения.
2. Сначала упрощаем логику, потом украшаем.
3. Меньше вариантов в моменте = быстрее выбор.
4. “Дорого и современно” достигается чистотой и пропорциями, а не количеством эффектов.

## 3. Цвет и контраст
1. Цвет используется экономно и осмысленно:
   - нейтральные тона для базы,
   - акцент для действий,
   - семантика для статусов.
2. Цвет не единственный носитель смысла (дублируем текстом/иконкой/формой).
3. Поддерживаем стабильный контраст текста и интерактивных элементов.
4. Ограничиваем число конкурентных акцентных цветов на одном экране.

## 4. Layout и spacing
1. Единая шкала отступов (кратность базовому шагу).
2. Отступ показывает отношение элементов:
   - ближе = связаны,
   - дальше = новый смысловой блок.
3. Выравнивание системное, а не “на глаз”.
4. Избегаем “воздуха ради воздуха”: компактно, но не тесно.
5. Отступы от краев экрана одинаковы внутри одного контекста.

## 5. Типографика
1. Текстовые роли фиксированы: title, section title, body, hint, meta.
2. Заголовки описательные и понятные вне контекста.
3. Текст не превращается в сплошные полотна:
   - дробим на смысловые блоки,
   - используем подпункты и маркировку.
4. Для длинного body-текста целимся в комфортную длину строки (ориентир: 40-80 символов).
5. Типографика строит иерархию, а не декорирует интерфейс.

## 6. Copywriting в интерфейсе
1. Тексты кнопок описывают результат действия (“Сохранить теги”, а не “ОК”).
2. Лейблы полей и подсказки максимально конкретны.
3. Ошибки объясняют:
   - что пошло не так,
   - как исправить,
   - что будет после исправления.
4. Убираем размытые и маркетинговые формулировки из критических сценариев.

## 7. Формы
1. Спрашиваем только необходимое.
2. Предпочитаем одноколоночный поток для мобильного UX.
3. Лейблы всегда видимы, placeholder не заменяет label.
4. Ошибки показываются рядом с полем и сразу после валидации.
5. Первичное действие формы визуально одно и явно доминирует.

## 8. Кнопки и действия
1. На одном экране один primary button на один контекст задачи.
2. Secondary и tertiary не конкурируют с primary.
3. Destructive actions:
   - визуально отделены,
   - подтверждаются,
   - не стоят рядом с частыми безопасными действиями.
4. У каждой кнопки есть состояния: `default / hover / active / disabled / loading`.

## 9. Состояния интерфейса
1. Каждая зона имеет сценарии: `loading / empty / error / success`.
2. Состояния не ломают layout и не дергают контент.
3. Skeleton/placeholder должен повторять будущую структуру контента.
4. Пользователь всегда понимает, происходит ли действие и что делать дальше.

## 10. Навигация и когнитивная нагрузка
1. Минимизируем число шагов до целевого действия.
2. Не заставляем запоминать скрытые условия.
3. Возврат назад не разрушает контекст без явной причины.
4. Overlay-потоки должны сохранять состояние базового экрана.

## 11. Mobile first, iOS/PWA
1. Учитываем safe-area, системные панели, динамический viewport.
2. Touch-цели удобны для пальца, жесты не конфликтуют.
3. Ключевые сценарии проверяются отдельно на iOS Safari/PWA.

## 12. Визуальный характер (ориентир shadcn)
1. Простая геометрия, чистые поверхности, аккуратные радиусы.
2. Минимум декоративного шума; приоритет — контент и действие.
3. Единый стиль компонентов по всему продукту.
4. Контролируем “премиальность” через ритм, типографику, тонкий контраст, а не через перегруз эффектами.

## Анти-паттерны (запрещено по умолчанию)
1. Два и более равносильных primary action в одном контексте.
2. Случайные отступы и “точечные” размеры без системы.
3. Слабый контраст текста/контролов.
4. Ошибки без инструкций по исправлению.
5. Лишние карточки/плашки, которые не добавляют смысла.

## Quality gate для PR (кратко)
1. Экран проходит разделы `1-12` этого документа.
2. Есть проверка `loading/empty/error`.
3. Нет регрессии ключевого пользовательского пути.
4. Визуальные изменения консистентны с остальными экранами волны.
